关于 DevOps 的一些思考

DevOps 是一种理念，是一种文化。它的宗旨是在一个产品的生命周期内，所有的角色都能以互相合作的方式尽早的参与到生命周期管理，并通过不断迭代去完成一个几乎完美的产品，使其更符合用户的需求。本文旨在对 DevOps 这一理念进行一些思考和规划一些可实践标准。

关于 DevOps 的一些思考

一、什么是 DevOps？

1、关于 Agile

🌈 我们之所以要在一开始提及Agile 这个概念是因为 DevOps 从某种程度上可以视作是 Agile 的实践与发展。

Agile，即敏捷项目，是一种以用户的需求进化为核心，采用迭代、循序渐进的方法进行软件项目开发的模式。 迭代是整个 Agile 理论的核心，在软件项目构建初期即会被分成多个子项目，子项目之间互相联系但也可以独立运行，子项目在软件开发阶段分别完成，并进行集成测试部署，迭代对象不再是软件整体，而变成了其中的子项目，因此在软件的生命周期中，始终处于可使用的状态。

由于这种模式很符合当前移动互联网背景下，通过更好的使用户满意来提高产品竞争力的愿景，使之能被广大的软件开发团队所认可，并且形成了具体的实践方法，如 Scrum。

在《敏捷宣言》中，定义了敏捷开发的十二条指导原则:

  • 最重要的目标: 通过尽早和持续交付有价值的软件来使客户满意。
  • 欣然面对需求的变化，即使这种变化出现在开发阶段的后期，敏捷流程可以驾驭变化，我们用变化来获取客户竞争力。
  • 持续交付可工作的软件，从几星期到几个月，时间尺度越短越好。
  • 在项目的整个生命周期内，业务人员和开发人员必须通力合作。
  • 围绕斗志高昂的人进行软件开发，给开发者提供适宜的环境，满足他们的需要，并相信他们能够完成任务。
  • 不论团队的内外，传递信息效果最好、效率最高的方式就是面对面的交谈。
  • 可工作的软件是进度的首要衡量标准。
  • 敏捷过程倡导可持续开发。责任人、开发者和用户要能够共同维持其步调稳定延续。
  • 对于卓越技术和良好设计的不懈追求有助于提高敏捷的能力。
  • 以简洁为本 —— 尽可能减少不必要的工作量是一门艺术。
  • 最好的架构、需求和设计都源自自我组织（自驱）的团队。
  • 团队需要定期的总结如何进一步提高效率，然后相应地调整自己的行为。

2、关于 DevOps

🌈 DevOps 是 Agile 思想的进一步扩展，它将开发和运营团队的合作看做实现软件产品持续交付的必要手段。

DevOps = 通过产品开发和运营团队的合作来支持软件产品（IT服务）的迭代开发和持续交付的能力。 用下面这张图可以较好的体现这一点:

1）DevOps 的三步工作法

1. 流动原则

🚩 它加速了从开发、运维到交付给客户的流程。

目的：建立Dev和Ops的共同目标，合力创造更高的价值 —— 更好更快的满足客户的诉求，以此来达到持续运营的业务目的

核心思想：“持续交付” —— 交付是一个 “从左到右” 的工作流，通过“流水线”的方式可以实现持续构建、测试和集成、确保代码和基础设施始终处于可部署的状态，并通过各环境的不断验证，最终使得提交的代码都可以安全的部署到生产环境

2. 反馈原则

🚩 它使我们能建设出更安全可靠的工作体系。

目的：促进团队构建安全、可靠、不断反馈的工作体系

核心思想: “质量保障的左移” —— 复杂的系统不可避免的会存在错误，可以通过采取安全措施和监控手段，确保在质量问题出现之前，快速发现和处理错误；质量保障是一个不断 “从右到左” 前置的过程，从源头保障质量永远胜于在结尾做验收检查，团队中的每个角色都要需要在整个项目生命周期内通力合作，并对最终生产发布的质量负责

3. 持续学习与实验原则

🚩 它打造出一种高度信任的文化和一种科学的工作方式，并将对组织的改进和创新作为日常工作的一部分。

目的：在前两个原则的基础上，建立起一个内驱的、不断自我完善并乐于分享的团队文化，在提升单兵能力的同时更要形成团队做战力

核心思想：“需求不只是外来的” —— 要实现自我完善，就要通过不断的内部挖掘可能的优化点，将内部优化的需要也作为需求输入加入产品规划。对于前瞻性的优化点进行 POC 实践加以验证，将实践的有效成果进行团队内的分享，在提升单兵能力的同时形成团队作战力

2）DevOps 是一种组织文化、流程、工具的结合体

对于 DevOps 来说，工具、流程、组织文化 这三个要素同样重要，缺一不可，缺少工具支持就无法实现”高速”，缺少流程就无法维持”高效”，缺少组织文化支持会让团队成员间无法”团结一致”共同完成目标。

二、DevOps 能带来什么改变？

从最直接的概念来看，就是软件产品或IT服务的交付方式发生了变化。例如，正常情况下交付一个有十个功能的软件产品需要十个月的时间，在 Agile/DevOps 的交付模式下就会变成一个月交付一个功能，总共交付十次。如下图:

![交付模式的改变](https://blog-images.tareya.cn/blog_images/交付模式的改变 .png)

在交付的过程中，用户可以提前得到最终产品或服务的一部分进行实际体验（测试质量团队也应当被视为用户的一部分），从而可以尽快的将反馈传达给需求管理团队和产品研发团队，然后根据得到的反馈进一步迭代、交付，在这周而复始的过程中产品将会变得越发完善，就仿佛一个人很难在没有工具借助的情况下画出一个完美的圆，但是在通过不断的画圆，将这些不完美的圆层层叠加，最终就会得到一个接近完美的圆。

这种模式的改变可以为企业带来很高的业务价值。因为它最大限度的解决了用户需求描述和产品功能之间的信息不对称问题。众所周知的产品生命周期包括从产品需求确立到产品设计，开发，生产及售后。

在传统模式下直到产品生产完成后，我们才能确认产品提供的服务或应用是否能真正的满足用户的需求，但是由于各种无法克服的原因，信息在传递的过程中会不断的被扭曲，被误解，导致最终产品和用户的期望总会有比较大的出入，如果需要重新改正的话，成本往往过于巨大导致用户不得不长时间忍受着不符合自己期望的产品。

特别是在移动互联网时代背景下，许多企业的商业模式也不断发生着改变，从 ToB 进一步拓展到 ToC 的业务模式，这也就意味着用户群体的扩大，我们上面所说的情况往往就会导致用户群体的流失，使产品失去竞争力。

而 DevOps 这种关注用户需求通过持续交付获取用户需求反馈，从而去完善产品，使之更贴合用户需求，使产品始终保持更高的竞争力，这是一种令人向往的商业价值的体现。

其他的，诸如成本节约、资源高效利用等价值，都是附带拓展给企业内部带来的红利。

三、对于 DevOps 的常见误区

1、DevOps 只适用于创业公司

2、DevOps 将取代 Agile

3、DevOps 与 ITIL 不兼容

1）关于 ITIL

ITIL 是信息技术基础架构库（Information Technology Infrastructure Library）的缩写。于20世纪80年代末，由英国中央计算机与电信局 CCTA 所创建，旨在满足将信息技术应用于商业领域的发展需求。现由英国商务部 OGC 负责管理维护，已更新到 ITIL 4，主要适用于IT服务管理（ITSM），是一个基于行业最佳实践的框架。它为企业的 IT 服务管理实践提供了一个客观、严谨、可量化的标准和规范。

4、DevOps 与信息安全合规活动不兼容

5、DevOps 意味着消除 IT 运维，即 “NoOps”

6、DevOps 只是 “基础设施即代码” 或自动化

7、DevOps 只适用于开源软件

四、转型 DevOps 需要做些什么？

DevOps 模式需要原本各自为战的部门在一种新的公司文化下更加紧密的合作，这就需要改变以个人部门利益为主的思维模式。

一般的企业习惯采用 KPI 的方式来衡量事情的进度，对于 DevOps 来说也是一样的，我们可以使用 DevOps 成熟度指标模型来引导对于 DevOps 的转型。

级别1·初始级（Initial Level） 在组织局部范围内开始尝试DevOps活动并获得初期效果；

级别2·基础级 （FundamentalLevel） 在组织较大范围内推行DevOps实践并获得局部效率提升；

级别3·全面级（ComprehensiveLevel） 在组织内全面推行DevOps实践并贯穿软件全生命周期获得整体效率提升；

级别4·优秀级（Excellent Level ） 在组织内全面落地DevOps并可按需交付用户价值达到整体效率最优化；

级别5·卓越级（Fabulous Level） 在组织内全面形成持续改进的文化并不断驱动DevOps在更大范围内取得成功。

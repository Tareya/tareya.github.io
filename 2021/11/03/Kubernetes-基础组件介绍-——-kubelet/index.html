Kubernetes 基础组件介绍 —— kubelet

在 K8S 的组件中，kubelet 是在每个 Node 节点上运行的 “节点代理”，接收并执行 Master 节点发来的指令，并实际管理 Pod 以及 Pod 中的容器。并且 kubelet 会通过 cAdvisor 来监控节点和容器的资源，并定期向 Master 节点汇报该节点的资源使用情况。

Kubernetes 基础组件介绍 —— kubelet

一、Kubelet 的工作原理

在 K8S 的组件中，kubelet 是在每个 Node 节点上运行的 “节点代理”。它默认监听 10250 端口，接收并执行 Master 节点发来的指令，并实际管理 Pod 以及 Pod 中的容器。每个 kubelet 进程都会在 API Server 注册所在 Node 节点的信息，其注册的表示一般为主机名(hostname)、覆盖主机名的参数或公有云驱动的特定逻辑。kubelet 会通过 cAdvisor 来监控节点和容器的资源，并定期向 Master 节点汇报该节点的资源使用情况。

二、Kubernetes 的资源体系

1、资源需求和资源限制

在 K8S 的资源体系中存在两个概念 requests 和 limits，用于为容器分配 cpu 和 memory 资源，定义在 .spec.containers.resources 中。一般来说，limits 会大于 requests。

  • requests: 即资源需求，用于保障容器可正常运行的最低保障，是容器调度的判断依据之一
  • limits: 即资源限制，容器可以分配到的最大资源配额，通过这种硬性限制来控制容器资源的使用

2、QoS Classes 分类

Kubernetes 在创建 Pod 时会为其指定 QoS 类，它会影响到 Pod 的调度和驱逐策略。

1）Guaranteed

确保类型，此类 Pod 具备最高的优先级。如果 Pod 中所有容器的 .spec.containers.resources 中的 limits 和 requests 都相等且不为0，则这个 Pod 的 Qos 类就是 Guaranteed。需要注意的是，如果一个容器只指明了limit，而未指明 request，则表明 request 的值等于 limit 的值，也就符合 Guaranteed 的要求。

4）QoS Classes 优先级排名

🚩 QoS 类的优先级为: Guaranteed > Burstable > Best-Effort

当节点出现资源紧缺时，kubelet 会根据 QoS 的优先级驱逐 Pod，最先被驱逐的就是 BestEffort。在 Burstable类型下，已占用量与需求量比值大的先被驱逐（如，pod1 request 内存为 512Mi，已占用内存 500Mi； Pod2 request 内存为 1024Mi，已占用 512Mi 内存，那么会先驱逐 pod1）

3、可压缩资源和不可压缩资源

K8S 中对于系统资源可以分为两种属性：

  • 可压缩资源: 例如 CPU 循环、Disk I/O、网络带宽，都是可以被限制和被回收的，对于一个 Pod 来说，可以降低这些资源的使用量而不需要去驱逐 Pod
  • 不可压缩资源: 例如内存、磁盘空间，对于不可压缩资源来说，如果资源不足，就无法继续申请资源，需要通过驱逐节点的 Pod 来进行资源回收

kubelet 回收的资源主要是内存和磁盘两种，回收的基本方式如下:

  • 磁盘回收: 主要通过删除已终止的容器和未使用的镜像
  • 内存回收: 主要通过终止正在运行的 pod

4、预分配资源

和容器一样，在没有做任何限制的情况下，K8S 的 Pod 能够使用所在 Node 节点的所有可用资源。这事实上是一个问题，因为我们一般会在节点上运了许多系统守护进程（系统内核、驱动、kubelet 等），它们也是要消耗资源的，除非我们为这些守护进程预留出一定的资源，否则它们就会与 Pod 一起加入到资源争夺的行列，很可能会直面节点资源短缺的问题，最严重的情况将无法维持系统的稳定运行。

1）节点可用计算资源量

Kubernetes 节点上的 Allocatable 被定义为 Pod 的可用计算资源量。调度器在调度 Pod 时时不会超额申请 Allocatable 的，目前可支持的资源为 cpu、memory、ephemeral-storage。

2）预留资源

在 kubelet 中，可以为两类系统守护进程预留资源，一个是操作系统预留资源 system-reserved，另一个为 K8S 系统预留资源 kube-reserved。

1. system-reserved

system-reserved 用于为诸如 sshd、udev 等系统级守护进程预留资源（cpu、内存、磁盘）；并且为系统内核预留内存资源（kernel 所使用的内存并不包括在 Pod 中）；以及为用户登陆会话预留资源（systemd 体系中的 user.slice）

除了 Allocatable 相关的 cpu、memory 、ephemeral-storage 参数外，pid 参数可用于为OS系统进程预留指定的 进程 ID。

system-reserved 的配置方式有两种，一种为 kubelet 命令行参数 --system-reserved，另一种为 kubelet 的 yaml 配置文件参数 systemReserved。

三、Kubernetes 节点压力驱逐

1、什么是节点压力驱逐？

节点压力驱逐事实上就是 kubelet 主动终止 Pod 以回收节点资源的过程。kubelet 会监控集群节点的 CPU、内存、磁盘空间和文件系统的 inode 等资源。 当这些资源中的一个或者多个达到特定的消耗水平， kubelet 可以主动地使节点上一个或者多个 Pod 失效，以回收资源防止枯竭。

在节点压力驱逐期间，kubelet 会将所选的 Pod 的 PodPhase 设置为 Failed，这回终止 Pod。Pod 的信息残留反馈为 Evicted 状态，如下图:

2、节点压力驱逐的机制

kubelet 会根据 驱逐信号，驱逐条件、监控间隔 来作出驱逐决定。

1）驱逐信号

驱逐信号是特定资源在特定时间点的当前状态。kubelet 使用驱逐信号，通过将信号和驱逐条件进行比较，来做出驱逐决定，驱逐条件就是节点上应该可用资源的最小量。kubelet 可使用的驱逐信号如下:

驱逐信号 取值方式
memory.available memory.available:=node.status.capacity[memory]-node.stats.memory.workingSet
nodefs.available nodefs.available:=node.stats.fs.available
nodefs.inodesFree nodefs.inodesFree:=node.stats.fs.inodesFree
imagefs.available imagefs.available:=node.stats.runtime.imagefs.available
imagefs.inodesFree imagefs.inodesFree:=node.stats.runtime.imagefs.inodesFree
pid.available pid.available:=node.stats.rlimit.maxpid-node.stats.rlimit.curproc

每个驱逐信号都支持百分比或是字面值，kubelet 采用的是信号有关的总量的百分比的方式来进行判断。

1. memory.available

memory.available 的值来自 cgroupfs，而不是像 free -m 这样的系统工具，这里系统工具在容器内实际上是起不到效果的（因为容器根本上在底层是公用的系统资源）。而 CGroup 则是将容器进行了分层次区分，每个容器都会有其唯一的 CGroup 根节点，kubelet 会判断该根节点的可用资源来计算 memory.available。

具体的计算方法如此脚本memory-available.sh。kubelet 在其计算会排除了 inactive_file（即非活动 LRU 列表上基于文件来虚拟的内存的字节数）， 因为它假定在压力下内存是可回收的。

2. nodefs

nodefs 表示节点的主文件系统，适用于本地磁盘、emptyDir、日志存储等，它关注的是本地文件系统的磁盘空间 nodefs.available 和 inode 数 nodefs.inodesFree。

3. imagefs

imagefs 表示的是容器运行时，存储容器镜像和容器可写层，是将本地文件系统根据 CGroup 分层后的文件系统，它关注的是该层文件系统的磁盘空间 imagefs.available 和 inode 数 imagefs.inodesFree。

2）驱逐条件

kubelet 的驱逐条件形式为 [eviction-signal][operator][quantity]，其中:

  • eviction-signal: 即上述的驱逐信号
  • operator: 表示关系运算符，比如大于小于等
  • quantity: 表示触发驱逐条件的指标，比如 1Gi。其表示形式要与 Kubernetes 的形式匹配，支持数值形式或百分比（10%）

使用方式如下，以节点的总内存为 10Gi 为例，我们想将触发条件设置为1 Gi，则可以将驱逐条件定义为以下两种形式中的一种:

  • memory.available<10%
  • memory.available< 1G
1. 软驱逐条件

在设置软驱逐条件时，必须指定一个宽限期，否则会报错。在超过宽限期之前，kubelet 不会立刻驱逐 Pod

在设置宽限期的同时还可以指定 Pod 终止的宽限期上限，用于在触发软驱逐时优雅的终止 Pod。当两者同时被设置，并且满足软驱逐阈值时，kubelet 会使用两者间较小的那个

软驱逐的配置参数:

  • eviction-soft: 驱逐条件列表，驱逐条件如 memory.available<1.5Gi，如果驱逐条件持续较长时间超过指定的宽限期，就会触发 Pod 驱逐
  • eviction-soft-grace-period: 驱逐宽限期列表，如 memory.available=1m30s，用于定义软驱逐在正式驱逐 Pod 前必须保存多久，与驱逐条件绑定
  • eviction-max-pod-grace-period: 在满足软驱逐条件而终止 Pod 时，使用的最大允许宽限期（单位为秒）
2. 硬驱逐条件

硬驱逐条件没有宽限期，当达到硬驱逐的阈值时，kubelet 会立即杀死 Pod 以回收资源（如内存），避免资源耗尽

硬驱逐的配置参数:

  • eviction-hard: 驱逐条件列表，驱逐条件如: ``memory.available<1Gi`

kubelet 默认的硬驱逐条件:

  • memory.available<100Mi
  • nodefs.available<10%
  • imagefs.available<15%
  • nodefs.inodesFree<5%（Linux 节点）

3）驱逐监控间隔

kubelet 根据其配置的 housekeeping-interval（默认为 10s）间隔来评估驱逐条件

3、节点压力状况

kubelet 会报告节点状况以反映节点处于压力之下，其与驱逐信号的对应关系如下:

节点压力表现 驱逐信号 描述
MemoryPressure memory.available 节点上的可用内存已满足驱逐条件
DiskPressure nodefs.available、nodefs.inodesFree、imagefs.available或imagefs.inodesFree 节点的根文件系统或镜像文件系统的磁盘空间或 inode 已满足驱逐条件
PIDPressure pid.available (Linux) 节点上的可用进程标识符已低于驱逐条件

通过配置 kubelet 的参数 nodeStatusUpdateFrequency 可以调整节点压力条件的更新间隔，默认为 10s

4、节点存储资源不足

关于节点存储资源默认的驱逐触发条件为:

  • nodefs.available<10%
  • imagefs.available<15%

1）驱逐机制

  • 当 imagefs 达到阈值时，kubelet 会删除节点上所有未使用的容器镜像来清理磁盘空间，以此回收存储资源

  • 当 nodefs 达到阈值时，kubelet 会拒绝新调度到该节点的 Pod 运行，并向 APIServer 通知 DiskPressure condition。然后会尝试删除已死亡的 Pod 和容器来回收磁盘空间，如果经过清理后仍低于阈值，kubelet 就会开始驱逐 Pod（不会考虑QoS优先级，而是根据 nodefs 的实际使用量大小进行驱逐排序，如果驱逐的事 DaemonSet，kubelet 会阻止其重启，直到 nodefs 的可用量大于阈值）

2）举例说明

有如下 Pod，其存储资源使用情况如下:

Pod Name Pod QoS nodefs usage
A Best Effort 800M
B Guaranteed 1.3G
C Burstable 1.2G
D Burstable 700M
E Best Effort 500M
F Guaranteed 1G

当 nodefs 的使用量超过阈值时，kubelet 会根据 Pod 的 nodefs 使用量来对 Pod 进行排名，首先驱逐磁盘使用量最多的 Pod。驱逐顺序排名如下：

Pod Name Pod QoS nodefs usage
B Guaranteed 1.3G
C Burstable 1.2G
F Guaranteed 1G
A Best Effort 800M
D Burstable 700M
E Best Effort 500M

5、节点内存资源不足

关于节点内存资源默认的驱逐触发条件为:

  • memory.available<100Mi

1）驱逐机制

当触发阈值时，kubelet 会向 APIServer 通知 MemoryPressure condition，此时 kubelet 将不再接受新的 QoS 为 BestEffort 的 Pod 在节点运行，并会按照以下顺序来驱逐 Pod:

  • Pod 的内存使用量是否超过了 request 指定的值
  • 根据 priority 排序，优先级低的 Pod 最先被驱逐（保证 Guaranteed 不会最先被驱逐）
  • 比较它们的内存使用量与 request 指定的值之差（container_memory_usage_bytes 指标值与 container_resource_requests_memory_bytes 指标值的差，如果有多个容器，则为所有差值的总和）

2）举例说明

有如下 Pod，其内存资源使用情况如下:

Pod Name Pod QoS Memory requested Memory limits Memory usage
A Best Effort 0 0 700M
B Guaranteed 2Gi 2Gi 1.9G
C Burstable 1Gi 2Gi 1.8G
D Burstable 1Gi 2Gi 800M
E Best Effort 0 0 300M
F Guaranteed 2Gi 2Gi 1G

当内存的使用量超过阈值时，kubelet 会根据 Pod 相对于 request 的内存使用量来对 Pod 进行排名，首先驱逐相对使用量最多的 Pod。驱逐顺序排名如下：

Pod Name Pod QoS Memory requested Memory limits Memory usage 内存相对使用量
C Burstable 1Gi 2Gi 1.8G 800M
A Best Effort 0 0 700M 700M
E Best Effort 0 0 300M 300M
B Guaranteed 2Gi 2Gi 1.9G -100M
D Burstable 1Gi 2Gi 800M -200M
F Guaranteed 2Gi 2Gi 1G -1G

⚠️ 注意: 当内存资源不足时，kubelet 在驱逐 Pod 时，只会考虑 requests 和 Pod 的内存使用量，并不会考虑 limits。

6、节点 OOM 时

1）驱逐机制

正常情况下，kubelet 会通过驱逐条件杀死 Pod 来回收内存资源，但是由于 kubelet 是默认 10s 抓取一次 cAdvisor 的监控数据，所以是有可能在驱逐回收之前出现 OOM 情况的。这个时候 Node 节点就会依赖内核的 oom_killer 来响应。

kubelet 根据 Pod 的 QoS 会为每个容器设置一个 oom_score_adj 值，如下:

QoS oom_score_adj
Guaranteed -997
BestEffort 1000
Burstable min(max(2, 1000 - (1000 * memoryRequestBytes) / machineMemoryCapacityBytes), 999)

OOM killer 会杀掉 oom_score_adj 值最高的容器，如果有多个容器的 oom_score_adj 值相同，就会杀掉内存使用量最多的容器（其实是因为内存使用量最多的容器的 oom_score 值最高），以此来降低系统压力。但是与 Pod 驱逐不同的是，OOM 杀死的容器，kubelet 是可以根据其 RestartPolicy 重启它的。

omm_score 的计算公式如下:

oom_score = 容器使用的内存占系统内存的百分比 + oom_score_adj

2）举例说明

假设某节点运行着 4 个 Pod，且每个 Pod 中只有一个容器。每个 QoS 类型为 Burstable 的 Pod 配置的内存 requests 是 4Gi，节点的内存大小为 30Gi。每个 Pod 的 oom_score_adj 值如下所示：

Pod Name Pod QoS oom_score_adj
A Best Effort 1000
B Guaranteed -998
C Burstable 867（根据上面的公式计算）
D Best Effort 1000

当调用 OOM killer 时，它首先选择 oom_score_adj 值最高的 BestEffort 容器（1000），这里有两个容器的 oom_score_adj 值都是 1000，OOM killer 最终会选择内存使用量最多的容器，然后如果还需要继续回收内存资源，就会驱逐 Burstable 容器。

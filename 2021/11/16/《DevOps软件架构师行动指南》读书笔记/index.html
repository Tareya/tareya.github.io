《DevOps软件架构师行动指南》读书笔记

DevOps 是在保证质量的前提下，提供的一整套从开发，测试到生产运维的持续交付和管控方法论。在整个过程中需要实现自动化，可视化，流水线式作用，同时将质量管控无缝嵌入其中。
———《DevOps软件架构师行动指南》-伦思·拜斯

《DevOps软件架构师行动指南》读书笔记

一、DevOps 概述

1、DevOps 对部署的要求

  • 部署对系统的变更时（通常是代码形式），质量很重要
  • 这个定义要求交付机制也是高质量的
  • 我们认为有两个周期很重要
    • 一个是开发人员提交新开发的代码的时间，这标志着基本的开发过程结束，部署工作开始
    • 另一个是把代码部署到生产环境的时间
  • 我们的定义是目标导向的
  • 最后，定义中说明的目标并不限于DevOps用于测试和部署的实践

2、最佳实践要求

  • 从需求的角度把运维人员视为首要干系人，这些实践符合定义中高质量方面的要求。运维人员有一套与日志和监控相关的需求
  • 让开发人员更多地负责相关事件的处理。新部署的内容在开始一段时间由开发人员主要负责，之后由运维人员主要负责
  • 强制推行所有人使用的部署过程，包括开发人员和运维人员。确保更高的部署质量
  • 使用持续部署
  • 在开发诸如部署脚本这样的基础设置代码时，也应遵守一套与应用程序代码相同的实践

3、为什么使用 DevOps

  • 运维人员能力有限
  • 减少运维复杂度
  • DevOps运维采用一种不同的方法。它们的方法把以前由运维完成的很多任务都自动化了，并让开发人员承担部分剩余的工作，通过这种方法可以减少对专业运维人员的需求

4、小团队优势

  • 可以快速做出决定
  • 人少比人多更容易组成一个关系密切的群体
  • 人们在小团队面前比在大团队面前更容易表达意见
  • 一个小团队需要完成一小部分代码。围绕着一组微服务架构是一种好方法，它能够把这些小任务打包在一起并减少显式配合 — 所以我们将开发团队的输出成果称为”服务”

5、团队角色

  • 服务所有者

    • 即能与其他干系人交流，又能与团队的其他成员交流

  • 可靠性工程师

    • 必须出色的排查故障、诊断问题
    • 必须深入理解服务的内部，这样才能修复故障或采用临时解决方案
    • 需要逐步成长为有能力的开发人员，因为他们需要编写高质量的程序并且以自动化的方式实现不断重复发生的诊断、迁移和修复工作

  • 看门人角色

    • 在部署流水线中，决定将服务进入下一步的手动角色
    • 决定服务的某个版本或服务的某个部分是否能够通过”门”并进入下一步

  • DevOps 工程师

    • 负责 DevOps 工具链中使用的各种工具的维护和保养
    • 这个角色可以是个人、团队或组织层面
    • 选择开发团队使用的配置管理工具的哪个发型版是DevOps工程师职责的一部分
    • 为开发团队定制工具并监控开发人员是否正确使用也是 DevOps 工程师的职责
    • 以自动化的方式实现开发和部署流水线是DevOps 的固有职责

  • 一个人可以承担多个角色，一个角色也可以由多人承担，角色的分配取决于个人能力、个人的工作负荷以及这个角色所需的能力和工作量

6、团队协作

  • 为什么需要团地协作
    • 不同团队开发的各部分能够在一起工作
    • 避免重复工作
  • 协作的形式
    • 直接的
    • 间接的
    • 持久的
    • 短暂的
    • 同步的
    • 异步的
  • 人工协作
    • DevOps 人工过程取自敏捷过程，目的是用于持久性不高的高带宽协作
    • 站立会议和信息发射源是人工过程协作机制的两个例子
  • 自动化协作
    • 自动化团队协作机制，用自动化的方式完成重复单调的任务，加快发现并报告错误的速度
    • 目标之一是尽可能快地向开发人员提供反馈
  • 跨团队的协作方式
    • 与干系人和客户的上游协作
    • 与运维人员的下游协作
    • 与其他开发团队的交叉协作
  • DevOps目标的一部分是通过使用隐性和不经常的协作代替直接的协作来实现的

7、推行 DevOps 的障碍

  • 文化冲突
    • 开发人员的工作职责和激励机制是做出变更（发布新代码），运维人员的工作职责和激励机制是保持稳定（不多变更），由于这种原因往往会导致”混沌之墙”的出现
  • 组织风险
    • 一个特定组织所担忧的风险取决于他们的活动领域，对一些组织来说，出现问题所带来的风险比尽快推向市场更重要
    • 在监管领域运作的组织（金融、医疗保健、公用事业服务）有他们必须遵守的规则，如果违反了运营规则，就会面临处罚，有可能还是很严重的处罚
    • 在成熟，行动缓慢的领域中运作的组织（汽车或建筑行业）有很长的前置时间
    • 某些组织的客户在切换到其他供应商（supplier）时成本很高，例如企业资源计划系统，客户因为担心运行的稳定性而不愿意冒险

二、云即平台

1、核心

  • 按需自助
  • 可计量

2、虚拟化平台的弊端

  • 由于一个虚拟化与一个物理机上的其他虚拟机共享资源，所以虚拟机之间可能会有性能干扰
  • 当家在虚拟机时，取决于底层物理基础设施和需要动态加载的其他软件，可能 有时间和可靠性方面的不确定性

3、PAAS 与 Ops 类比

  • 某种程度上，PASS 与传统 Ops 部门提供的一些服务相类似，Ops 部门通常接管基础设施层的管理并向开发团队提供一组环境选项来托管他们的系统，开发团队可以在这些环境中进行选择

4、正态分布

  • 诸如计算、读文件或者接收本地消息这样的简单请求都有接近正态分布

5、长尾分布

  • 复杂的请求，如大量的 Map-Reduce 任务，大型数据库的搜索或启动虚拟实例，将会有长尾这样的偏态分布
  • 耗时过长的请求应当被当作故障，防止长尾的一个机制是取消耗时过长的请求

6、数据一致性

  • 在分布式系统中，通过引入锁来维护一致性，这些锁控制访问单个数据项的顺序。但锁定数据项会带来访问这些数据项的延迟，因此，铲射高了各种不同的方案来维护一致性并减少由锁引起的延迟

  • 根据称为CAP（Consistency: 一致性、Availability: 可用性、Partition tolerance: 分区容错性）原理的理论结果，不可能同时拥有完全可用、一致并分区的数据。最终一致性意味着当改变数据项时，分布、分区和复制的数据即使没有立即一致，在一段时间后，副本最终将会变得一致

7、多环境支持

  • 在开发、测试和部署过程的各阶段有多个环境并不是云独有的特性，但简单地创建和迁移环境的能力时云独有的特性，就如克隆新实例一样容易

三、IT 服务和运维

1、IT 职能

  • 服务台的运维

    • 服务台的员工负责处理所有的事件和服务请求，并作为所有问题的第一级处理者

  • 技术专家

  • IT 服务的日常经验

    • 这些包括周期性和重复性的维护运维、监控、备份以及设施管理

2、软硬件供给

  • 防止内部研发的软件延期时 DevOps 的一个驱动力
  • 硬件标准化

3、业务连续性

  • 恢复点目标（RPO，Recovery Point Objective）

    • 当灾难发生时，对数据丢失的最长容忍时间是多少？如果每小时备份数据一次，则恢复点目标为1小时，因为丢失的数据可能是自上一次备份后的增量数据

  • 恢复时间目标（RTO，Recovery Time Objective）

    • 当灾难发生时，不能提供服务的最长容忍时间是多少？例如，如果一种方案需要10分钟来获取在另一个数据中心的备份，再需要5分钟来实例化新的服务器以便使用这些备份的数据，则恢复时间目标为15分钟

4、服务设计原则

  • 标准化合同
  • 松耦合
  • 抽象
  • 可复用性
  • 自治
  • 无状态
  • 可发现性
  • 可组装性

5、服务移交

  • 服务移交包含服务的实施和交付两个阶段。DevOps 和持续部署要求服务移交的交付部分高度自动化，这样它才能够处理高频率的移交，并提供更好的质量控制
  • 变更类型
    • 标准变更（例如，经常发生和低风险的变更）
    • 常规变更
    • 紧急变更

6、监控职责区分

  • 开发团队分析他们自己开发的单系统监控的结果
  • 包括基础设施的多系统监控则由运维团队负责
  • 运维团队还负责需要一个或多个开发团队合作处理的事故上报

7、持续服务改进

  • 主要焦点是要在 IT 服务和业务需求之间达成一致 —— 不管这些需求是否已发生变化还是维持不变
  • 如果这些需求已经发生变化，则对 IT 服务的期望变化可以关注范围、功能或服务级别协议
  • 如果业务需求相同，则可以扩展 IT 服务来更好的支持它们，但对它们的改进也可以专注在提升效率上

8、ITIL 与 DevOps

  • 如果认为 ITIL 是过于”重量级”尔不适合 DevOps 过程，那么这个观点是短视的，并且这个观点将导致要再走过那些 ITIL 框架中所试图解决的”坑”
  • ITIL 服务移交和 DevOps 方法之间的一个差别在于 ITIL 假设非常大的发布包（需要小心计划、变更管理等）是可行的，而典型的 DevOps 场景中较常遇到的是高频率的小的发布包

四、整体架构

1、模块与组件

  • 模块是一个具有一致性功能的代码单元
  • 组件则是一个可执行单元
  • 编译器或解释器将模块转化为二进制文件，构建器将二进制文件转换为组件
  • 开发团队直接开发模块。组件是开发团队开发模块的结果，所以也可以说开发团队开发组件，但我们应该清楚组件的开发是开发团队的一个间接活动
  • 微服务: 微服务架构将一组提供少量功能的服务集合整合到一起，这些系统的整体功能来自于多个服务

2、服务通信

  • 客户端和服务器之间的通信协议可以是任何远程通信协议，如 HTTP、RPC、SOAP 等。这些服务可以提供 RESTful 接口，也可以不提供
  • 远程通信协议应该是不同服务间通信的唯一方法。服务提供的接口细节则需要跨团队的协作

3、无状态服务

  • 如果实例中不保留状态，那么销毁它就没什么代价，经过一段冷却期后，实例不再接收新的请求并响应请求，这个实例就可以被销毁了
  • 无状态服务的另一个优势是消息可以发送给服务的任何实例，这样将有助于在实例之间进行负载均衡

4、指派与分配

  • 工作指派
    • 一个团队可以在多个模块上工作，但让多个开发团队工作在相同模块上需要这些开发团队之间大量的协作工作，因为协作会花费时间，所以更简单的一种结构是将单个团队的工作打包成模块并再开发模块的接口，以允许不同团队开发的模块相互操作
  • 分配
    • 每个组件（即微服务）以独立的可部署单元存在

5、可靠性问题

  • 数量不多的团队协作
    • 数量有限的团队间的协作可能导致开发客户端的团队与开发服务的团队在接口语义方面产生歧义
    • 当涉及大量微服务和真实外部资源时，频繁执行集成测试和端到端测试代价高昂
    • 使用一种叫作消费者驱动契约（CDC，Consumer Driven Contract）的测试实践可以缓解这个问题
  • 环境正确性
    • DevOps 中的一个重要趋势是像管理应用程序代码一样引入版本控制和测试来管理设置环境的代码和配置参数
  • 服务实例故障
    • 幂等（Idempotent）是服务的一种机制，允许服务被相同的输入重复调用，同时产生相同的输出 —— 换而言之，没有产生错误
    • 如果服务失败，客户端可以执行代替操作

6、可修改性

  • 使服务具有可修改性意味着让可能的变化简单化，并减少变化带来的连锁反应
  • 需要具备可修改性的原因
    • 识别可能的变化
      • 服务执行所在的环境
      • 与你的服务交互的其他服务的状态
      • 你的服务所使用的第三方软件和库的版本
    • 减少连锁反应
  • 使服务具有更灵活的可修改性的方法
    • 封装可能变化的影响部分
    • 封装可能引起变化的连锁反应的交互部分

7、团队的亚马逊规则

  • 所有团队通过服务接口开放他们的数据和功能
  • 团队必须通过接口与其他团队通信
  • 不允许其他形式的服务间/团队间的通信
    • 没有直接链接
    • 不能直接访问其他团队的数据存储
    • 没有共享内存模式
    • 没有后面
    • 允许的唯一通信是通过网络上的服务接口调用
  • 不用关心他们（其他服务）使用了何种技术
  • 所有服务接口，无一例外，都要在一开始就设计称可以外部化的，团队必须计划和设计能够将接口对外部开发者开放
  • 服务级别协议绑定了客户端和服务端，如果客户端的需求超过了服务级别协议中承诺的工作负荷，则响应速度慢就会变成客户端问题，而非服务端问题

8、微服务方案

  • 在需求规格说明时考虑运维关注的部分
  • 被开发的系统整体结构应该是一个小的、独立的服务组成的集合
  • 对客户端和其他需要的服务，每个服务都是不可信的
  • 需要定义和理解团队角色
  • 服务需要注册到本地的注册表/负载均衡器
  • 服务必须周期性地刷新他们的注册信息
  • 服务需要给客户端提供服务级别协议
  • 服务要设计成无状态且短暂的
  • 如果服务必须维持状态，则它应该在外部持久存储上维护状态数据
  • 如果服务所依赖的其他服务发生了故障，则需要备用方案
  • 服务有防御检查，以便拦截来自客户端的错误输入或其他服务的输出
  • 外部服务、环境信息、第三方软件和库的使用都应该本地化（例如，他们要求通过特定于那个外部服务、环境信息、外部软件或库的模块）

五、构建和测试

1、对于基础设施的要求

  • 团队成员可以并行地在系统的不同版本上工作（版本库）
  • 一个团队成员开发的代码不会在无意中覆盖另一个团队成员开发的代码（版本库，代码合并规范）
  • 如果一个团队成员突然离开团队，工作不会丢失（安全性）
  • 团队成员的代码易于测试（编码规范）
  • 团队成员的代码可以容易地与同一个团队的其他成员编写的代码集成在一起（编码规范）
  • 一个团队开发的代码可以容易地与其他团队开发的代码集成在一起（通过 api 交互集成）
  • 系统的集成版本可以容易地部署在不同环境中（例如，测试环境、预发布环境(staging)、生产环境）
  • 系统的集成版本可以在不影响系统生产版本的情况下容易且完整地测试（灰度）
  • 可以对系统最近部署的新版本进行密切监控（服务监控、链路监控）
  • 一旦代码放入生产环境，如果出现问题，能够找到旧版本的代码（回滚方案）
  • 万一出现问题代码可以回滚（回滚方案）

2、CI/CD 流水线

  • 定义 持续集成 的一种方法是在一个阶段与下一个阶段之间有一个自动触发器，直到集成测试。即，如果构建成功，则触发集成测试。如果不成功，则通知对失败负责的开发人员。
  • 持续交付 定义为，使用自动触发器，直到预发布环境，并进行用户验收测试、性能测试等环节。
  • 持续部署 意味着到最后一步（即部署到生产环境）也是自动化的。在将服务部署到生产环境之后（蓝绿/灰度），会密切监控一段时间，然后再将其升级到正式生产环境。

3、可追溯性

  • Git 与 Cms
    • 基础设施即代码
    • 把所有的东西都放入版本控制或配置管理系统，让你能够在任何一个地方重新创建准确的环境，从本地环境到生产环境。这不但对追踪问题很有帮助，而且能够在新环境中快速且灵活地部署应用程序
  • 允许可追溯性的第一个选择就是把起标示作用的标签与打包的应用程序关联起来
  • 另一个选择是把生产环境中每个机器的起源放到一个外部配置管理系统中（CMDB）

4、环境

  • 本地: 开发人员的笔记本、台式机/工作站
  • 开发: 开发服务器，也称为沙盒
  • 集成: 持续集成 (CI，Continuous Integration) 构建目标，或用于开发人员测试副作用
  • 测试/质量保证 (QA，Quality Assurance) : 用于功能、性能测试、质量保证等
  • UAT: 用户验收测试
  • 预发布/生产前: 生产环境的镜像
  • 生产环境/实时: 服务于终端用户/客户

5、横切关注点

  • 测试框架（测试实例）
    • 测试框架是一组软件及配置的测试数据，通过在各种条件下运行它，并对他的行为以及输出进行监控，以便测试一个程序单元。测试框架是必须的，以便自动测试
    • 测试框架的一个关键特征是会产生报表，并且至少能识别出哪个测试失败了
  • 负面测试
    • 测试系统行为是否符合预定也是很重要，为这个目的所做的测试统称为负面测试
  • 回归测试
    • 回归测试是在通过最初的测试后维护并重新运行测试的根本原因。
    • 在对系统现有的功能及非功能领域进行改进、打补丁或配置变更等变更后，力求发现新的软件缺陷
  • 环境拆解
    • 当一个环境不再用于某个特定目的时，如预发布测试，就应该将它拆解
    • 拆解环境的一个原因是释放与此环境相关的资源，另一个原因是避免无意间与这些资源进行交互

6、版本控制与分支

  • 核心功能
    • 能够通过环境识别源代码的不同版本
    • 不同开发人员之间共享代码修订
    • 记录从一个版本到下一个版本之间谁做了修改
    • 记录修改范围
  • SVN
    • CVS 和 SVN 是集中式解决方案，其中每个开发人员都从中央服务器上检查代码，然后将变更提交回这台服务器（不高可用）
  • GIT
    • 每个开发人员都有包含所有内容的 Git 存储库的一个本地克隆 (副本)
    • 提交是在本地存储库完成的
    • 一系列变更可以通过中央服务器进行同步，其中服务器的变更与本地存储库同步 (git pull)
    • 本地变更可以提交给服务器上使用 (git push)
  • 分支管理
    • 两个问题
      • 分支可能太多，你不知道特定任务应该在哪个分支上完成
      • 合并两个分支可能是困难的
    • 替代方案
      • 分支结构的一个替代方案是让所有开发人员都在主干上直接工作，与使用分支相比，它需要更频繁的操作
      • 所有开发工作都在一个主干上的问题是，会在部署流水线中的新功能中引入不完整或未测试的代码
      • 为了解决这个问题，提出了功能开关
    • 功能开关（功能门）
      • 功能开关允许你持续地交付新发布，这些版本可能包含未完成的新功能，但不会影响到应用程序，因为这些新功能还处于关闭状态
      • 教训1: 不要重复使用开关名称
      • 教训2: 尽可能快地集成功能并去掉开关测试

7、配置参数简单化

  • 配置参数的数量应该控制在一个可管理的水平
  • 过多的配置参数通常会导致它们之间的复杂联系，多个参数之间的兼容设置集也只有几个软件配置专家才知道
  • 虽然灵活性是我们期望的目标，但是太复杂的配置意味着你基本上是在创建一个专属的编程语言

8、持续集成 (CI) 步骤

  • 持续集成服务器得到新提交的通知或者定期检查是否有新提交
  • 在检测到新的提交时，持续集成服务器可以检索它
  • 持续集成服务器运行构建脚本
  • 如果构建成功，则持续集成服务器运行自动化测试
  • 持续集成服务器把它的活动的结果提供给了开发团队（例如，通过内部网页或电子邮件）

9、用户验收测试/预发布/性能测试

  • 用户验收测试（UAT, User Acceptance Test）
    • 预期用户使用系统当前版本的用户界面所做的测试，可以根据测试脚本测试，也可以使用探索性测试方法
  • 自动化验收测试
    • 可重复的用户验收测试的自动化版本
    • 自动化验收测试在白天或夜间的少量时间能够完成更多可重复性的工作
  • 冒烟测试
    • 自动化验收测试的一个子集，用于快速分析提交的代码是否破坏了应用程序的某些核心功能
    • 一个密闭的流水线系统充满了烟雾，如果发生了泄漏，可以容易地检测到烟雾
  • 非功能测试
    • 对性能、安全、容量以及可用性等方面的测试
  • 早期发布的几种方式
    • 最传统的方法是 beta 发布
    • 金丝雀测试方法（灰度）
      • 把新版本的首先部署到几台服务器上，观察它们如何执行
    • A/B 测试
      • 与金丝雀相似，差别在于 A/B 测试的目的是，对于特定的业务的关键绩效指标（KPI）来说，确定哪个版本执行得更好
  • 现场测试（Live Testing）
    • 测试的另一种形式是干扰正在运行的系统，这种形式也叫做现场测试

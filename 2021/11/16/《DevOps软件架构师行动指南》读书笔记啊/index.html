《DevOps软件架构师行动指南》读书笔记

DevOps 是在保证质量的前提下，提供的一整套从开发，测试到生产运维的持续交付和管控方法论。在整个过程中需要实现自动化，可视化，流水线式作用，同时将质量管控无缝嵌入其中。
———《DevOps软件架构师行动指南》-伦思·拜斯

《DevOps软件架构师行动指南》读书笔记

一、DevOps 概述

1、DevOps 对部署的要求

  • 部署对系统的变更时（通常是代码形式），质量很重要
  • 这个定义要求交付机制也是高质量的
  • 我们认为有两个周期很重要
    • 一个是开发人员提交新开发的代码的时间，这标志着基本的开发过程结束，部署工作开始
    • 另一个是把代码部署到生产环境的时间
  • 我们的定义是目标导向的
  • 最后，定义中说明的目标并不限于DevOps用于测试和部署的实践

2、最佳实践要求

  • 从需求的角度把运维人员视为首要干系人，这些实践符合定义中高质量方面的要求。运维人员有一套与日志和监控相关的需求
  • 让开发人员更多地负责相关事件的处理。新部署的内容在开始一段时间由开发人员主要负责，之后由运维人员主要负责
  • 强制推行所有人使用的部署过程，包括开发人员和运维人员。确保更高的部署质量
  • 使用持续部署
  • 在开发诸如部署脚本这样的基础设置代码时，也应遵守一套与应用程序代码相同的实践

3、为什么使用 DevOps

  • 运维人员能力有限
  • 减少运维复杂度
  • DevOps运维采用一种不同的方法。它们的方法把以前由运维完成的很多任务都自动化了，并让开发人员承担部分剩余的工作，通过这种方法可以减少对专业运维人员的需求

4、小团队优势

  • 可以快速做出决定
  • 人少比人多更容易组成一个关系密切的群体
  • 人们在小团队面前比在大团队面前更容易表达意见
  • 一个小团队需要完成一小部分代码。围绕着一组微服务架构是一种好方法，它能够把这些小任务打包在一起并减少显式配合 — 所以我们将开发团队的输出成果称为”服务”

5、团队角色

  • 服务所有者

    • 即能与其他干系人交流，又能与团队的其他成员交流

  • 可靠性工程师

    • 必须出色的排查故障、诊断问题
    • 必须深入理解服务的内部，这样才能修复故障或采用临时解决方案
    • 需要逐步成长为有能力的开发人员，因为他们需要编写高质量的程序并且以自动化的方式实现不断重复发生的诊断、迁移和修复工作

  • 看门人角色

    • 在部署流水线中，决定将服务进入下一步的手动角色
    • 决定服务的某个版本或服务的某个部分是否能够通过”门”并进入下一步

  • DevOps 工程师

    • 负责 DevOps 工具链中使用的各种工具的维护和保养
    • 这个角色可以是个人、团队或组织层面
    • 选择开发团队使用的配置管理工具的哪个发型版是DevOps工程师职责的一部分
    • 为开发团队定制工具并监控开发人员是否正确使用也是 DevOps 工程师的职责
    • 以自动化的方式实现开发和部署流水线是DevOps 的固有职责

  • 一个人可以承担多个角色，一个角色也可以由多人承担，角色的分配取决于个人能力、个人的工作负荷以及这个角色所需的能力和工作量

6、团队协作

  • 为什么需要团地协作
    • 不同团队开发的各部分能够在一起工作
    • 避免重复工作
  • 协作的形式
    • 直接的
    • 间接的
    • 持久的
    • 短暂的
    • 同步的
    • 异步的
  • 人工协作
    • DevOps 人工过程取自敏捷过程，目的是用于持久性不高的高带宽协作
    • 站立会议和信息发射源是人工过程协作机制的两个例子
  • 自动化协作
    • 自动化团队协作机制，用自动化的方式完成重复单调的任务，加快发现并报告错误的速度
    • 目标之一是尽可能快地向开发人员提供反馈
  • 跨团队的协作方式
    • 与干系人和客户的上游协作
    • 与运维人员的下游协作
    • 与其他开发团队的交叉协作
  • DevOps目标的一部分是通过使用隐性和不经常的协作代替直接的协作来实现的

7、推行 DevOps 的障碍

  • 文化冲突
    • 开发人员的工作职责和激励机制是做出变更（发布新代码），运维人员的工作职责和激励机制是保持稳定（不多变更），由于这种原因往往会导致”混沌之墙”的出现
  • 组织风险
    • 一个特定组织所担忧的风险取决于他们的活动领域，对一些组织来说，出现问题所带来的风险比尽快推向市场更重要
    • 在监管领域运作的组织（金融、医疗保健、公用事业服务）有他们必须遵守的规则，如果违反了运营规则，就会面临处罚，有可能还是很严重的处罚
    • 在成熟，行动缓慢的领域中运作的组织（汽车或建筑行业）有很长的前置时间
    • 某些组织的客户在切换到其他供应商（supplier）时成本很高，例如企业资源计划系统，客户因为担心运行的稳定性而不愿意冒险

二、云即平台

1、核心

  • 按需自助
  • 可计量

2、虚拟化平台的弊端

  • 由于一个虚拟化与一个物理机上的其他虚拟机共享资源，所以虚拟机之间可能会有性能干扰
  • 当家在虚拟机时，取决于底层物理基础设施和需要动态加载的其他软件，可能 有时间和可靠性方面的不确定性

3、PAAS 与 Ops 类比

  • 某种程度上，PASS 与传统 Ops 部门提供的一些服务相类似，Ops 部门通常接管基础设施层的管理并向开发团队提供一组环境选项来托管他们的系统，开发团队可以在这些环境中进行选择

4、正态分布

  • 诸如计算、读文件或者接收本地消息这样的简单请求都有接近正态分布

5、长尾分布

  • 复杂的请求，如大量的 Map-Reduce 任务，大型数据库的搜索或启动虚拟实例，将会有长尾这样的偏态分布
  • 耗时过长的请求应当被当作故障，防止长尾的一个机制是取消耗时过长的请求

6、数据一致性

  • 在分布式系统中，通过引入锁来维护一致性，这些锁控制访问单个数据项的顺序。但锁定数据项会带来访问这些数据项的延迟，因此，铲射高了各种不同的方案来维护一致性并减少由锁引起的延迟

  • 根据称为CAP（Consistency: 一致性、Availability: 可用性、Partition tolerance: 分区容错性）原理的理论结果，不可能同时拥有完全可用、一致并分区的数据。最终一致性意味着当改变数据项时，分布、分区和复制的数据即使没有立即一致，在一段时间后，副本最终将会变得一致

7、多环境支持

  • 在开发、测试和部署过程的各阶段有多个环境并不是云独有的特性，但简单地创建和迁移环境的能力时云独有的特性，就如克隆新实例一样容易

三、IT 服务和运维

1、IT 职能

  • 服务台的运维

    • 服务台的员工负责处理所有的事件和服务请求，并作为所有问题的第一级处理者

  • 技术专家

  • IT 服务的日常经验

    • 这些包括周期性和重复性的维护运维、监控、备份以及设施管理

2、软硬件供给

  • 防止内部研发的软件延期时 DevOps 的一个驱动力
  • 硬件标准化

3、业务连续性

  • 恢复点目标（RPO，Recovery Point Objective）

    • 当灾难发生时，对数据丢失的最长容忍时间是多少？如果每小时备份数据一次，则恢复点目标为1小时，因为丢失的数据可能是自上一次备份后的增量数据

  • 恢复时间目标（RTO，Recovery Time Objective）

    • 当灾难发生时，不能提供服务的最长容忍时间是多少？例如，如果一种方案需要10分钟来获取在另一个数据中心的备份，再需要5分钟来实例化新的服务器以便使用这些备份的数据，则恢复时间目标为15分钟

4、服务设计原则

  • 标准化合同
  • 松耦合
  • 抽象
  • 可复用性
  • 自治
  • 无状态
  • 可发现性
  • 可组装性

5、服务移交

  • 服务移交包含服务的实施和交付两个阶段。DevOps 和持续部署要求服务移交的交付部分高度自动化，这样它才能够处理高频率的移交，并提供更好的质量控制
  • 变更类型
    • 标准变更（例如，经常发生和低风险的变更）
    • 常规变更
    • 紧急变更

6、监控职责区分

  • 开发团队分析他们自己开发的单系统监控的结果
  • 包括基础设施的多系统监控则由运维团队负责
  • 运维团队还负责需要一个或多个开发团队合作处理的事故上报

7、持续服务改进

  • 主要焦点是要在 IT 服务和业务需求之间达成一致 —— 不管这些需求是否已发生变化还是维持不变
  • 如果这些需求已经发生变化，则对 IT 服务的期望变化可以关注范围、功能或服务级别协议
  • 如果业务需求相同，则可以扩展 IT 服务来更好的支持它们，但对它们的改进也可以专注在提升效率上

8、ITIL 与 DevOps

  • 如果认为 ITIL 是过于”重量级”尔不适合 DevOps 过程，那么这个观点是短视的，并且这个观点将导致要再走过那些 ITIL 框架中所试图解决的”坑”
  • ITIL 服务移交和 DevOps 方法之间的一个差别在于 ITIL 假设非常大的发布包（需要小心计划、变更管理等）是可行的，而典型的 DevOps 场景中较常遇到的是高频率的小的发布包

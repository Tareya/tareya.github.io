1）场景介绍

使用 Nacos 作为服务注册中心，并通过 APISIX 的 nacos discovery 实现服务注册节点的动态获取。原本安装版本为 2.6.0，对于 nacos 值支持尚处于开发阶段。

2）版本选择

根据官方发布的服务 changelog，2.7.0 之后的版本对于 nacos discovery 有了一定的升级，故选择的版本需时 2.7.0 之后的版本。

CHANGELOG

目前官方 dockerhub 提供的最新稳定版本镜像为 2.8.0 ，故本次版本，选择使用 2.8.0 的版本，当然也可以使用官方提供的方式编译最新版本的镜像，看个人需求。

apisix 官方镜像

apisix-dashboard 官方镜像

apisix-ingress-controller 官方镜像

3）查看官方提供的 chart

🚩 官方提供的 chart 使用的版本与我们期望的版本不完全相符，但是整体的资源结构不会有太大的区别，也就是说我们可以基于官方提供的 chart 去制作我们自己需要的 chart，然后再通过 helm 进行纳管。

1、准备 chart

4）识别依赖组件

从官方 chart 的的目录结构可以看出，部署 apisix 同时还会有 etcd、dashboard、ingress-controller 三个依赖组件，基本和非容器化部署是一样的。etcd 作为数据存储，dashboard 提供 UI 界面，而 ingress-controller 则相当于 openresty。

1. etcd集群

etcd 只是作为数据存储来说，它的版本其实并不需要过多考虑，我们不需要对其进行更改。当然，也可以使用 K8S 本身的 etcd 集群，但是不推荐这么做。

2. apisix-dashboard

dashboard 根据上文可知，最新的镜像版本为 2.7.1 ，这是我们需要更改的。

3. apisix-ingress-controller

ingress-controller 实际上就是 ingress，其使用的 api group 是随 K8S 版本的，因此需要根据实际的 k8S 版本修改ingress 的版本。官方给出的参数解释如下:

ingress version

By default apisix-ingress-controller watches the Ingress resources in api group networking/v1, however, if your Kubernetes cluster is prior to v1.19, you need to change the ingress watching version.

If your Kubernetes version is older than v1.14, then:

Or if your Kubernetes version is between v1.14 and v1.19, try below:

本文使用的 K8S 环境为 v1.18.18 版本，其网络 api 组为 beta 版本，故 ingress 版本需选用 networking/v1beta1，我们需要在 chart 中定义，或则在发布时进行参数定义。
